# CapyBoard 上線評估與待你拍板的問題

> 日期：2026-07-05｜依 E2E 實測（見 `EVIDENCES/E2E_2026-07-05.md`）
> 對象問題：**要不要換框架/引擎？離「上架遊戲商城」還差多少？**

---

## TL;DR（先給結論）

1. **技術品質是好的。** build 過、40 單元測試全綠、0 console error、引擎架構乾淨（pure class + 單一真相）。以「一個做得完整的 side project」來說，它已經完成度很高。
2. **不需要換遊戲引擎（Unity / Godot / Phaser）。** CapyBoard 是回合制、非即時、無物理、無 60fps 繪圖需求的「桌遊設計工具」，React + DOM 完全夠用。換引擎＝整包重寫、零收益。
3. **建議把建置工具從 CRA 換成 Vite**（不是換引擎，是換 build 工具）——CRA（react-scripts）自 2023 起官方已停止維護，這是遲早要還的技術債，但**成本低、可延後**，不是上架前的擋路石。
4. **真正離「上架」遠的不是技術，是「產品定位」+「產品層功能」。** 目前它是一個**只跑在單一瀏覽器、無帳號、無雲端、無金流、無音效、編輯器/遊玩頁不支援手機**的本機工具。這些才是差距。
5. **最關鍵的是一個定位問題：你要上架的是「這個工具」，還是「用這個工具做出來的某一款遊戲」？** 兩者是完全不同的產品與商城。這題你不回答，後面所有估算都無意義。見下方「Q1」。

## 一、要不要換框架 / 引擎？

### 結論：引擎不換，build 工具建議換（可延後）

| 選項 | 我的判斷 | 理由 |
|------|----------|------|
| 換成 Unity / Godot / Cocos（遊戲引擎） | ❌ 不建議 | 這類引擎是為即時、物理、精靈動畫、跨平台打包而生。CapyBoard 是回合制文字/籌碼互動，用不到；換過去等於重寫且失去現有 web 即開即用的優勢。 |
| 換成 Phaser / PixiJS（2D 遊戲庫） | ❌ 不建議 | 同上，且會把現在乾淨的 React 元件化 UI 打掉重練。 |
| 保留 React，**把 CRA 換成 Vite** | ✅ 建議（可排在上架後） | React 程式碼幾乎不動，換掉停維護的 CRA、build 更快、生態更新。半天～1 天工。 |
| 保留 React + CRA 現狀 | 🟡 短期可接受 | 能 build 能跑，但 CRA 已 EOL，長期有安全更新與相依風險。 |

> 一句話：**你的痛點不在「引擎不夠力」，而在「這是個工具、不是個商品」**。換引擎解決不了任何一個真正的上架缺口。

## 二、離「正式產品上線」還差多少？（分層 gap）

我把差距分成三層。**技術債很小，產品層很大。**

### A. 技術/工程層（小，1–2 週級距）
- [ ] CRA → Vite 遷移（去除 EOL 相依）
- [ ] 補 `@types/jest` 讓 `tsc` 全綠（目前僅測試檔型別報錯）
- [ ] 加 E2E/CI 自動化（目前只有單元測試；本次 E2E 是手動跑）
- [ ] 錯誤邊界（ErrorBoundary）+ localStorage 損毀/超額（quota）的防護

### B. 使用體驗 / 完成度層（中，數週級距）
- [ ] **手機響應式**：遊玩頁三欄硬排、編輯器固定畫布 + react-dnd 不支援 touch → 手機幾乎不能用。要嘛做 RWD，要嘛明講「桌機工具」。
- [ ] **品牌一致性**：全站 UI 叫「桌遊大師」，但專案/repo 叫 CapyBoard，**畫面上完全沒有水豚 / Capy 元素或 logo**。商品化前要統一命名與品牌識別。
- [ ] **暖色設計準則的違規點**：Onboarding 圖示與提示框、玩家序號圓標、當前玩家高亮、規則 Pxx 優先序標記、空白遊戲「＋」都還是**冷藍色**，違反 CLAUDE.md 的暖色蠟筆準則。要一次掃清。
- [ ] **遊戲手感（juice）**：目前只有擲骰有動畫，**完全無音效**、無得分/勝利的動態回饋。商品化的遊戲通常需要這層。
- [ ] 無 zones 的遊戲（如骰子冒險）執行頁只有純文字面板，視覺偏陽春。

### C. 產品 / 商業層（大，這才是「上架」的主體）
- [ ] **後端 / 帳號系統**：目前 100% localStorage。換瀏覽器/清快取＝作品全失，無法跨裝置。商品化幾乎必須有帳號 + 雲端儲存。
- [ ] **分享 / 社群 / 內容市集**：現在「分享」只能手動匯出/匯入 JSON。若定位是 UGC 平台，需要作品發佈、瀏覽、遊玩他人作品的機制。
- [ ] **金流 / 商業模式**：免費？訂閱？單機買斷？商城抽成？——完全未定。
- [ ] **多人連線**：目前是單機同螢幕輪流。線上多人是另一個 phase 的大工程（先前記錄也列為 Phase 2）。
- [ ] **法務/上架要件**：隱私政策、服務條款、年齡分級、（若上 App Store/Google Play）平台審查與帳號、（若上 Steam）Steamworks 整合。
- [ ] **i18n**：資料有 `en` 欄位但 UI 文案全硬編中文，無翻譯系統。跨區上架要補。

# 決策記錄（2026-07-05 已拍板）

> 以下為你回答後定案，取代上方問題清單，作為後續開發依據。

## 產品定位（Q1 定案）
CapyBoard 不是單機工具，而是一個 **線上多人 UGC 桌遊平台 + 虛擬經濟 + 授權/房間 DRM**。核心願景：
- 使用者用簡約編輯器設計「可玩」的桌遊；**遊玩畫面要有美術呈現、不能太粗糙**（編輯畫面可簡約）。
- 遊戲可打包分享（用**自製規格的通訊協定/封包格式**）：可上架商城，或免費分享。
- **免費分享前提**：分享者必須是該遊戲作者；作者可生成**有限數量 key** 讓親朋好友取得使用權。
- 玩家可上架自製遊戲，收取**虛擬貨幣**；虛擬貨幣可購買商城內新遊戲或其他東西。
- **房間 DRM**：擁有使用權者可開房間邀他人同玩，但客人只有**該房間單次訪問權**；離開後不能自建同款遊戲的房間，除非自己也購入該遊戲。

## 已定參數
| 項目 | 決定 |
|------|------|
| 目標平台（Q2） | **Steam（桌機）＋ 手機（App Store / Google Play）＋ Web** 三者都要 |
| 目標裝置（Q3） | 桌機 + 手機**兩者都要**（必須投資 RWD + touch 拖放） |
| 商業模式（Q4） | 主程式**免費**；遊戲需自製或到商城購買 |
| 虛擬貨幣 | **真錢充值、不可提現**（作者收幣不能換現金） |
| 第一個里程碑（Q7） | **遊玩頁美術升級**（單機遊玩體驗先做到「不粗糙」） |
| 美術資源 | 以**免費 AI 生圖**可達到的程度為基準來設計美術工程（主題包/圖層） |
| 品牌（Q6） | **全面水豚化 CapyBoard**（吉祥物 + logo + 命名鋪進 UI，統一暖色家庭風） |

## 框架/引擎最終判斷（依上述定案更新）
**依舊不換遊戲引擎，反而更確定要以 React 為單一核心，用 web-wrapper 打三平台。**
- 三平台（Steam / 手機 / Web）若用原生引擎＝維護三套；用 **React 一套核心 + Capacitor（手機）+ Tauri 或 Electron（Steam/桌機）** 可**一份程式碼打三平台**，這正是不換引擎的最大戰略價值。
- 「遊玩頁美術升級」選的是**免費 AI 生圖等級**（非動態粒子/轉場的商業級），所以 **React/DOM + CSS 動畫 + AI 圖片主題包** 完全夠，**不需要 Canvas/WebGL/遊戲引擎**。
- 建議仍把 CRA → **Vite**（Capacitor/Tauri 生態對 Vite 支援最好），此步驟在動手前先做掉最省事。
- 線上多人房間需要**後端 + realtime netcode（WebSocket）**，這是最大的新工程，但屬中後期里程碑，不影響第一刀。

## 建議里程碑順序（我的排法，供你調整）
1. **M6 遊玩頁美術升級（第一刀，現在做）**：主題包系統（背景/token/卡面/勝負演出圖層）＋音效＋暖色/水豚化＋清冷藍違規＋無 zone 遊戲的視覺化。以「一份 AI 圖如何餵進遊戲」為設計核心。
2. **M7 手機/RWD + touch 拖放**：遊玩頁 RWD、編輯器 touch 後端。
3. **M8 建置現代化**：CRA → Vite，為跨平台打包鋪路。
4. **M9 帳號 + 雲端存檔 + 後端地基**。
5. **M10 自製封包格式 + 授權 key 系統（DRM 基礎）**。
6. **M11 商城 + 虛擬貨幣（真錢充值/不可提現）+ 金流/IAP（注意 Apple/Google 30% 抽成）**。
7. **M12 線上多人房間 + 單次訪問權控制（realtime netcode）**。
8. **M13 三平台打包上架（Capacitor / Tauri / Web）+ 法務頁（ToS/隱私/虛擬貨幣條款/分級）**。

> 註：M11 的真錢充值 + Apple/Google 內購 30% 抽成，是整個商業模式最需要早點想清楚的財務點；雖排在中段，但建議你先心裡有數。
